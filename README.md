# Adaptive Traffic Signal Control System

An AI-powered adaptive traffic signal controller that uses YOLOv8 vehicle detection and real-time lane-level demand estimation to dynamically allocate green-light durations across a four-phase intersection.

Unlike fixed-time controllers, every green interval is sized to match actual observed traffic — heavier lanes get more time, empty phases are skipped entirely, and the controller enters a full all-red idle state when no vehicles are present.

---

## Dataset

This project uses traffic surveillance footage from the AI City Challenge 2022 dataset.

- **Name:** AI City Challenge 2022 — Track 1: City-Scale Multi-Camera Vehicle Tracking
- **Source:** https://www.aicitychallenge.org/

The dataset provides real-world traffic camera footage captured from multiple viewpoints across urban road networks. For this project, selected videos are treated as independent approaches of a logical four-way intersection:

| Role            | Video File  |
|-----------------|-------------|
| North Approach  | `north.avi` |
| South Approach  | `south.avi` |
| East Approach   | `east.avi`  |
| West Approach   | `west.avi`  |

Each feed is processed independently — YOLOv8 detects vehicles, ROI polygons assign them to lane groups, and the resulting demand values are aggregated by the adaptive controller to drive signal timing decisions.

## Why This Dataset

The AI City Challenge 2022 Track 1 dataset contains real-world urban traffic footage captured from multiple cameras and varying viewpoints. This makes it suitable for evaluating vehicle detection, traffic density estimation, and adaptive traffic signal control strategies under realistic traffic conditions — including varied lighting, occlusion, and mixed vehicle types — without relying on synthetic or simulated data.

## Important Note on Signal Visualization

The traffic signal states displayed by this system are generated by the adaptive controller and are independent of the traffic lights visible in the source videos.

The AI City Challenge videos were recorded under real-world traffic conditions using the intersection's original fixed signal timings. Those original signal states are not used or read by this controller in any way.

Instead, the system:

1. Detects and classifies vehicles using YOLOv8.
2. Estimates lane-level traffic demand using ROI-based vehicle counts.
3. Computes phase demand values aggregated across all four approach feeds.
4. Dynamically determines signal timings using the adaptive traffic control algorithm.

As a result, the signal state shown in the dashboard will differ from the traffic light visible in the source footage. This is expected and reflects the controller's simulated decision-making process rather than the original intersection signal timing.

## How It Works

```
north.avi ──┐
south.avi ──┼──► YOLOv8 Detection ──► ROI Assignment ──► Lane Demand
east.avi  ──┤
west.avi  ──┘
                                                              │
                                               Phase Demand Aggregation
                                                              │
                                          Adaptive Traffic Controller
                                                              │
                                     GREEN / YELLOW / RED Signal Decisions
                                                              │
                                          2×2 Composite Dashboard Output
```

Four independent video feeds each represent one approach of a logical four-way intersection. The feeds can come from different cameras or locations — the controller only consumes the demand numbers produced by each feed, not the raw video.

## Signal Phases

The controller uses a four-phase protected-turn model with a fixed round-robin order.

| Phase | Movements Served              | Lane Groups Used                                  |
|-------|-------------------------------|---------------------------------------------------|
| P1    | North–South Straight + Right  | `north_straight_right` + `south_straight_right`   |
| P2    | North–South Left Turn         | `north_left` + `south_left`                       |
| P3    | East–West Straight + Right    | `east_straight_right` + `west_straight_right`     |
| P4    | East–West Left Turn           | `east_left` + `west_left`                         |

Phase order is always P1 → P2 → P3 → P4 → P1 → …  
Traffic demand affects **green duration** and **whether a phase runs**. It never changes the order.

## Adaptive Signal Logic

### Vehicle Weights

| Vehicle    | Weight |
|------------|--------|
| Car        | 1.0    |
| Bus        | 2.0    |
| Truck      | 2.0    |
| Motorcycle | 0.5    |

### Demand Formula

```
demand = (1.0 × cars) + (2.0 × buses) + (2.0 × trucks) + (0.5 × motorcycles)
```

### Green Time Allocation

```
green_time = min(MAX_GREEN, MIN_GREEN + k × demand)
```

Default values (`config.py`):

| Parameter      | Default |
|----------------|---------|
| `MIN_GREEN`    | 15 s    |
| `MAX_GREEN`    | 60 s    |
| `YELLOW_TIME`  | 5 s     |
| `DEMAND_FACTOR_K` | 1.0  |

**Examples at k = 1.0:**

| Demand | Green Time      |
|--------|-----------------|
| 5      | 20 s            |
| 20     | 35 s            |
| 50     | 60 s (capped)   |

### Phase Skipping

If a phase has zero demand it is skipped with no green and no yellow — the controller moves directly to the next valid phase.

### Idle Mode

If all four phase demands are zero, the controller enters **IDLE** and holds all signals RED. When traffic reappears, the controller resumes the round-robin sequence starting immediately after the **last completed phase** — it does not restart from P1 and does not jump to the highest-demand phase.

Example: if P3 was the last active phase and new demand arrives on P1 and P4, the controller checks P4 → P1 → P2 → P3 in order. P4 gets green first even if P1 has higher demand.

## ROI Calibration

Each approach uses its own pair of lane-group polygons calibrated to that camera's angle and resolution. The shipped coordinates are **placeholders** and must be replaced before running on real footage.

### Lane Groups per Approach

```
<approach>_left            — dedicated left-turn lane
<approach>_straight_right  — through and right-turn lane(s)
```

### Calibration Steps

**Step 1 — Extract a still frame**

```bash
python3 -c "
import cv2, sys
cap = cv2.VideoCapture(sys.argv[1])
ok, f = cap.read()
cv2.imwrite('calib_frame.png', f)
" videos/north.avi
```

**Step 2 — Record pixel coordinates**

Open `calib_frame.png` in any image viewer and hover over the four corners of each lane region. Note the pixel (x, y) coordinates.

**Step 3 — Update `config.py`**

```python
"north": {
    "video": "videos/north.avi",
    "rois": {
        "north_left":           [(x1,y1), (x2,y2), (x3,y3), (x4,y4)],
        "north_straight_right": [(x1,y1), (x2,y2), (x3,y3), (x4,y4)],
    },
},
```

Polygons can have more than 4 vertices for curved lanes.

**Step 4 — Verify**

Run `python3 main.py` and confirm the drawn polygon outlines align with the actual lane markings. Repeat for each of the four approaches.

### Lane Assignment Point

Vehicle positions are tested against ROI polygons using the **bottom-center** of each bounding box:

```python
cx = (x1 + x2) / 2
cy = y2
```

This approximates the vehicle's contact point with the road surface, which is more reliable than the bounding box center for overhead and angled cameras.

## Project Structure

| File                  | Responsibility                                                        |
|-----------------------|-----------------------------------------------------------------------|
| `config.py`           | All parameters: video paths, per-approach ROIs, phases, timing        |
| `detector.py`         | Shared YOLOv8 wrapper — one model instance runs across all four feeds |
| `roi_manager.py`      | Per-approach point-in-polygon assignment of detections to lane groups |
| `demand_calculator.py`| Weighted lane demand, cross-approach merging, phase aggregation       |
| `traffic_light.py`    | RED / YELLOW / GREEN state per approach and movement                  |
| `controller.py`       | Round-robin adaptive FSM: GREEN → YELLOW → next phase / IDLE         |
| `dashboard.py`        | Controller panel overlay + per-tile signal badge on each approach     |
| `main.py`             | Four-feed loop, 2×2 composite display, output video writer            |
| `test_logic.py`       | 50 synthetic unit tests — no camera or YOLO dependency required       |

## Running

### Configure video paths

Edit the `APPROACHES` dict in `config.py`:

```python
APPROACHES = {
    "north": {"video": "videos/north.avi", "rois": { ... }},
    "south": {"video": "videos/south.avi", "rois": { ... }},
    "east":  {"video": "videos/east.avi",  "rois": { ... }},
    "west":  {"video": "videos/west.avi",  "rois": { ... }},
}
```

### Start the system

```bash
# Use paths from config.py
python3 main.py

# Override individual paths at the command line
python3 main.py --north videos/north.avi \
                --south videos/south.avi \
                --east  videos/east.avi  \
                --west  videos/west.avi
```

A **1280×720 composite window** (four 640×360 tiles) opens showing all feeds simultaneously. Each tile is labelled with its approach name and carries a live signal-state badge showing the RED / YELLOW / GREEN state of its left and straight+right movements.

The **dashboard panel** in the top-left of the composite shows:

- Current phase and signal state
- Remaining green / yellow time
- Idle resume anchor (which phase the round-robin will continue from)
- Per-phase demand breakdown with skipped phases noted
- Per-approach demand and vehicle class counts

Press **`q`** to stop. If one feed ends before the others, its tile goes black and the remaining feeds continue until all are exhausted.

The annotated composite is saved to `outputs/composite_output.mp4` unless `SAVE_OUTPUT_VIDEO = False` in `config.py`.

### Run the unit tests

```bash
python3 test_logic.py
```

Runs 50 assertions covering ROI assignment for all four approaches, cross-approach demand merging, weighted demand formula, phase aggregation, adaptive green time, phase skipping, idle entry and exit (including the round-robin resume behaviour), first-startup edge case, round-robin wraparound, and status dict completeness. No video files or GPU required.

## Tuning

| Parameter            | Location    | Effect                                                                                          |
|----------------------|-------------|-------------------------------------------------------------------------------------------------|
| `DEMAND_FACTOR_K`    | `config.py` | How aggressively green time scales with demand. `k=1.0` gives demand 20 → 35 s, demand 50 → 60 s (capped). Raise to be more responsive to heavy traffic. |
| `MIN_GREEN`          | `config.py` | Floor green duration in seconds. No phase runs shorter than this.                               |
| `MAX_GREEN`          | `config.py` | Ceiling green duration in seconds.                                                              |
| `CONFIDENCE_THRESHOLD` | `config.py` | Lower to catch more vehicles on poor-quality footage; raise to reduce false positives.       |
| `DETECTION_FRAME_SKIP` | `config.py` | Run YOLO every Nth frame. `1` = every frame. Set to `3`–`5` if inference is too slow for real-time display; lane buckets carry over between skipped frames. |
| `TILE_W / TILE_H`    | `main.py`   | Resize the composite tiles to suit your monitor. Default is 640×360 per tile (1280×720 total). |
